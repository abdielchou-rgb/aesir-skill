# Æsir 故事织机 v4.0 架构建议

基于对全部交付物的理解和已有学术/行业方法论，对下一阶段产品架构的系统性建议。

---

## 一、当前架构的瓶颈

### 瓶颈 1：发散-收敛-延展是线性流水线

当前三个工作流 `diverge → converge → extend` 是一次性的线性流程。用户走完一轮要么"计入画布"要么"继续延展"。但最有趣的故事创意往往不是在一条线上走完发生的——而是**不同方向之间的交叉和碰撞**。

建议：画布中的多个故事单元之间应该允许**线索交叉引用**。用户说"A单元的未闭合线索3可以和B单元的主角结合起来吗？"——这是非线性创意最有价值的地方，当前系统不支持。

### 瓶颈 2：所有用户走同一个流程，没有用户模型

系统不对用户做区分。资深创作者和完全新手的交互路径应该不同：
- 新手：需要更多引导、更少的选项、发䋎时 3 条不是 5 条
- 有经验者：可以跳过驯化对话、直接使用 42 门禁、想要大纲而不是完整故事

建议：在使用 3-5 次后建立一个轻量的"用户画像"（偏好调性、偏好的故事长度、常选的冲突类型），后续流程自动适配。

### 瓶颈 3：画布只有"存"没有"查"

故事画布当前是一个单向积累的容器，缺乏检索和连接能力。用户积累了 20 个故事单元后，无法回答"哪个单元是关于地铁的""哪个单元里有红发女性角色"这类问题。

建议：画布中的每个单元自动标签化——基于文本内容抽取关键词（地名/角色名/核心物品），支持按关键词检索。不需要后端，前端可以做。

## 二、可以落地的 5 个改进

### 1. 分歧点不只在单元结尾出现

当前流程中，分歧点只在收敛后的"可发展的方向"中出现。实际上在一个故事单元的内部也可以有分歧。

建议：converge 在生成故事单元时，可以在中间某处标注"这里有一个分歧点"：
> (分歧点: 主角可以推开门, 也可以转身离开。选择A: 推开门 → 故事转向悬疑; 选择B: 转身 → 故事转向情感)

用户可以在单元之间做选择，而不是等到单元结束后才有选择权。

### 2. 跨单元联想

当用户画布中有多个故事单元时，系统应该主动发现它们之间的潜在关联：
- "你这个单元 B 里的'旧照片'和单元 A 里的'失踪的父亲'可能是同一条线索"
- "单元 C 的主角设定和单元 D 的配角可以合并成一个更有深度的人物"

这个不需要复杂的语义分析——基于关键词重叠做简单匹配即可。

### 3. 类型混搭

当前 diverge 只按单一类型发散。但好故事的创新往往来自类型混搭——"悬疑+爱情""仙侠+推理"。

建议：在 diverge 的 LLM prompt 中加入"如果多个类型可以混搭"的指令。用户说"悬疑"时，仍然以悬疑为主，但其中 1-2 条世界线可以混搭其他类型（如"悬疑+家庭伦理"）。

### 4. 收敛单元的可选粒度

当前 converge 固定输出 800-1200 字。但用户的场景和需求不同：
- 地铁上随手用：300-500 字够了
- 桌面端认真写：2000-3000 字更好
- 想写个大纲先看看走向：5-8 个节拍点就够了

建议：在收敛前问一句"这次要短篇、标准、还是大纲？"或者根据用户画像自动适配。

### 5. 审计结果驱动重写

当前 audit 只输出门禁结果，不驱动修改。用户可以拿着审计结果去手动重写，但没有"根据门禁自动修改"的能力。

建议：增加一个"按审计结果修改"的工作流。用户说"按审计意见改"时，系统根据门禁结果（如"触点不足""欲望不明"）对原文做针对性修改，保留原文风格。

## 三、更长远的方向

### 故事沙盒

不是线性流程，而是一个"沙盘"——用户反复做"发䋎→选择→收敛→调整"的循环，每次调整后的结果保留在画布中，随时可以退回到之前的某个决策点重新走。

### 协作叙事

多个用户的画布可以共享。A 延展了 B 的一个未闭合线索，系统自动标注"这条线索由 B 开启、由 A 延展"。

### 从故事单元到完整作品

故事画布中积累的多个单元可以自动检测"这些单元是否能拼成一个更大的故事结构"——基于未闭合线索的交叉程度和角色的重叠度。

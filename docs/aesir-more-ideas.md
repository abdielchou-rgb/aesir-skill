# 发散收敛的更多可能性思路

> 基于 2025-2026 年学术界和开源社区的最新探索，跳出"世界线→大纲→故事单元"的线性框架。

---

## 当前困境

目前的"发散→选择→收敛"本质上是一个**漏斗模型**——从 5 条世界线选 1 条，然后在这 1 条上深入。漏斗的问题是：你选了一条，其他 4 条就丢掉了。被丢弃的方向里可能埋着更好的种子。

有 3 个替代模型，可能比你提的"可能性大纲"更有想象力。

---

## 思路一：StoryBox 的涌现模型——不选线，养线

**来源：** StoryBox (AAAI 2026)
**核心思想：** 不让用户做选择，让多条世界线同时在后台运行，定期交融。

具体做法：

```
用户说"地铁上看到一个女孩在哭"

发散 5 条世界线 →
  线A: 陌生人善意(温暖治愈)
  线B: 她为什么哭(悬疑暗流)
  线C: 城市里哭泣的人(诗意思辨)
  线D: 看错了(黑色幽默)
  线E: 她哭的原因跟主角有关(激烈狂暴)

── 不选线 ──

每生成 500 字，停一下，问用户：
"目前 A 线和 B 线出现了一个交叉点——那个女孩和主角的过去有交集。你想让故事往哪个方向偏？"

── 线不是被选的，是被养的 ──

最终故事可能是 A+B 的融合，也可能是 B 吸收了 D 的调性。
用户做的不是"选哪条"，而是"往哪边偏"。
```

**优势：**
- 用户不需要在信息不足的时候做决策
- 融合线比单一线更有原创性
- 更接近真正的"涌现"

**代价：**
- 交互次数增加
- 需要更强的上下文能力

## 思路二：Athena 的多层叙事架构——7 层同时展开

**来源：** Themata-Claude-Scribe (GitHub 2026)
**核心思想：** 叙事不是线性的，是分层的。7 个层次同时展开，互相约束。

| 层次 | 内容 | 用户控制度 |
|------|------|----------|
| **原子事件** | 最基本的事件单元 | 低 |
| **场景** | 原子事件的组合 | 低-中 |
| **线索** | 跨越场景的叙事连接 | 中 |
| **角色弧** | 角色的变化轨迹 | 高 |
| **主题层** | 故事在讲什么 | 高 |
| **世界观层** | 故事的规则和边界 | 最高 |
| **元叙事层** | 这个故事为何被讲述 | 最高 |

在 Æsir 中的落地：

```
用户说想法 →
  → 系统生成 7 层骨架(不展示给用户)
  → 只问用户 2 个问题:
     "主题层: 你觉得这个故事是关于什么的?" 
     "角色弧: 主角到最后是变强了还是变脆弱了?"
  → 用户回答约束 7 层骨架
  → 收敛故事单元

用户不需要看大纲,大纲在后台。
用户只需要回答关于"意图"的问题——Dramatica 说这是最重要的。
```

**优势：**
- 用户像导演一样掌控高层次决策
- 低层次细节由系统自动填充
- 比大纲更轻量——用户只做选择题，不读文档

**代价：**
- 7 层模型对大多数用户太复杂，需要简化到 2-3 层
- 用户需要一定的叙事素养才能回答"主题层"问题

## 思路三：CreAgentive 的故事原型双图——同时发散情节和角色

**来源：** CreAgentive (arXiv:2509.26461)
**核心思想：** 把故事的"情节图"和"角色图"解耦，分别发散再融合。

具体做法：

```
用户说想法 →

  同时生成:
    情节图: 事件链 A → B → C → D(3-5条)
    角色图: 人物关系网络(谁认识谁,谁欠谁)

  让用户分别选:
    "你喜欢哪条事件链?"
    "你喜欢哪组人物关系?"

  系统把选中的情节图和角色图融合 →
    收敛为完整故事单元

  融合时自动检测冲突:
    "你选的事件链中有一个谋杀案,
     但你选的角色关系中没有受害者。
     要加一个受害者角色,还是换一条事件链?"
```

**优势：**
- 情节和角色分开选择，降低选择负担
- 交叉组合产生更多可能性（3 条情节 × 3 组关系 = 9 种组合）
- 融合过程本身就是创意的来源

**代价：**
- 需要更复杂的解析和融合逻辑
- 用户可能需要多次迭代才能找到好组合

## 思路四：MICE 嵌套感知——让用户选择"故事类型"而不是"故事方向"

**来源：** Orson Scott Card (MICE Quotient) + Mary Robinette Kowal (嵌套规则)
**核心思想：** 用户不需要选"方向"，只需要选"这是什么类型的故事"。

| 类型 | 核心问题 | 结构规则 |
|------|---------|---------|
| 世界型(Milieu) | "进入一个陌生世界又离开" | 进入→探索→离开 |
| 问题型(Idea) | "一个疑问被提出又被回答" | 问题→线索→答案 |
| 角色型(Character) | "一个人改变自己的角色" | 不满→选择→新身份 |
| 事件型(Event) | "世界失衡后恢复秩序" | 失衡→斗争→恢复 |

在 Æsir 中的落地：

```
用户说"地铁上看到一个女孩在哭"

系统问:
"你觉得这是个什么样的故事？
  A. 关于'她为什么哭'(问题型·悬疑)
  B. 关于'哭的人后来怎么样了'(角色型·成长)
  C. 关于'这个城市里哭泣的陌生人'(世界型·都市)
  D. 关于'这个地铁站出事了'(事件型·灾难)"

用户选 A(问题型·悬疑)

→ 系统按问题型的结构规则生成故事单元:
  问题提出 → 第一条线索 → 假答案 → 反转 → 真答案

→ 用户随时可以切换类型:
  "原来不是因为那件事,是因为这个人"→切换为角色型
```

**优势：**
- 用户不需要理解叙事结构，只需要选"这像什么故事"
- 每种类型有明确的结构规则，收敛结果更可控
- 类型混搭产生新可能（问题型+角色型嵌套）

**代价：**
- 需要用户对故事类型有直觉感受
- 类型切换时需要重新对齐上下文

## 我目前认为最好的方案：**思路二(7 层骨架简化版) × 思路四(MICE 类型选择)**

把两个思路合并：

```
用户说想法 →

  第一步：系统只问一个类型问题
    "你觉得这是个什么样的故事？"
    (提供 4 个选项,每个选项对应 MICE 的一种类型)
  
  第二步：系统只问一个意图问题
    "到最后,你希望主角_____?"
    (提供 2-3 个选项,对应角色弧的方向)

  第三步：收敛
    系统从前两步的答案推导出: 
    - 叙事结构(问题型=线索链,角色型=弧线)
    - 情感基调(主角变强/变脆弱)
    然后收敛为完整故事单元

  第四步：可选扩展
    用户说"这个方向还有别的可能吗？"
    系统保持类型不变,替换线索链中的关键事件
```

**为什么这样好：**
1. 用户只做 2 次选择题，不需要读任何文档
2. 类型和意图两个维度已经足够约束收敛方向
3. 收敛结果基于学术框架(MICE + Dramatica)，不是随机的
4. 用户用完一次后，对"故事类型"有了直觉认知，下次能更精确地表达需求

## 改动量评估

| 改动 | 工作量 | 说明 |
|------|--------|------|
| diverge 增加类型选项 | 小 | 在 LLM prompt 中加 MICE 分类逻辑 |
| converge 增加类型感知 | 中 | 根据用户选择的类型调整收敛的结构规则 |
| 当前代码兼容性 | 高 | 新增类型字段，不影响现有功能 |
| 对用户的影响 | 低 | 新增一个选择题，不会增加认知负担 |

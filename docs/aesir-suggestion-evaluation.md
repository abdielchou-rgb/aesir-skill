## 建议①：模糊想法先交流

**你的说法：** 用户说出模糊想法时，Bot 应该先跟用户交流再发散。

**学术依据：✔ 完全正确**

| 来源 | 为什么支持 |
|------|----------|
| **Wallas 五阶段模型**（1926） | 创意过程的"准备阶段"（Preparation）要求有意识地收集素材、定义问题空间。用户模糊时直接生成是跳过了准备阶段 |
| **Geneplore 模型**（Finke, 1992） | 生成（Generate）之前需要"前创意结构"——用户自己都没想清楚时，系统产生的世界线是盲目的 |
| **CHI 2025 Dreamsmithy** | 用户需要多轮控制感，单向输入-输出不够。对话建立信任感 |
| **Scriptnotes**（August+Mazin） | 场景必须有清晰处境——观众必须知道角色想要什么。在这之前，用户自己也未必知道 |
| **Cron 神经叙事学** | "没有情感就没有阅读"——没有先理解用户的情感锚点，生成的故事无法激活读者的神经卷入 |

**补充：3 个问题不是固定的，应该根据用户输入动态生成**

固定问"最让你在意什么？读者带走什么？参考作品？"对某些用户有用，但对另一些用户可能太抽象。更好的做法是：

- 如果用户说了一个具体的场景（"今天地铁上看到一个女孩在哭"）→ 问场景延伸："她哭的时候周围的人是什么反应？"
- 如果用户说了一个抽象主题（"我想写关于错过和重逢"）→ 问类型偏向："你觉得这是个遗憾的故事，还是还有机会的故事？"
- 如果用户说了一个类型词（"我想写个悬疑"）→ 问核心悬念："你心里想的那个秘密是什么级别的？"

建议在 system prompt 中增加"根据用户输入类型选择提问方向"的规则。

## 建议②：发散收敛遵循类型规律

**你的说法：** 应该遵循类型电影或类型文学的规律。

**学术依据：✔ 完全正确，且有明确的学术支撑**

| 来源 | 为什么支持 |
|------|----------|
| **Story Grid**（Coyne） | 每种类型有明确的"惯例"（conventions）和"必备场景"（obligatory scenes）。不满足这些用户会感觉"不对味" |
| **亚里士多德《诗学》** | 悲剧有特定的结构要求（完整的行动、一定长度、Peripeteia + Anagnorisis）——不同的类型有不同的诗学 |
| **McKee** | 不同类型有不同的核心价值和情感契约。悬疑的核心价值是"真相 vs 幻象"，爱情是"连接 vs 孤独" |
| **Snyder Save the Cat** | 不同类型在同一节拍位置上需要不同的内容——"坏人逼近"在爱情片里是误会升级，在动作片里是真正的威胁 |
| **网文行业** | 爽点分布因类型而异——逆袭流需要"降维打击"密集，情感流需要"情感共鸣"为主 |

**当前 Coze Bot 已经做到了什么：**

diverge 工作流的 LLM prompt 中已经加入了类型惯例（悬疑/爱情/成长/动作），世界线输出带`类型提示`字段。converge 的 SG5 结构是跨类型的通用骨架。

**可以再加强的地方：**

1. **类型调用的可视化：** 目前类型惯例是作为 prompt 文本嵌入的。更完整的做法是让用户明确说"我想写个悬疑"时，Bot 主动说"好的，悬疑类故事的惯例是：异常开场 → 线索释放 → 假答案 → 反转 → 真相。我来按这个节奏设计世界线。"

2. **类型混搭：** 很多好故事是类型混搭（悬疑+爱情、仙侠+悬疑）。用户说"我喜欢悬疑但也想有感情线"时，Bot 应该能够并行应用两组类型惯例。

## 建议③：收敛后应该是完整的故事单元

**你的说法：** 不是一个片段，而是一个有开头-中间-结尾的完整单元。

**学术依据：✔ 这是从"片段"到"故事"的关键跃迁**

| 来源 | 为什么支持 |
|------|----------|
| **亚里士多德** | 故事的最低条件就是"完整性"——有开头、中间、结尾。这是叙事的最基本定义 |
| **Todorov 三态模型** | 平衡→破坏→新平衡。一个"单元"就是一次从破坏到新平衡的完整周期 |
| **Pixar Story Spine** | 7 步骨架：很久以前→每天→有一天→因为这→因为这→直到终于→从那以后。收敛后的故事单元至少要走完"有一天→因为这→直到终于"这三步 |
| **网文行业** | 每一章/每一卷都是一个独立的信息释放单元——读完全章有获得感 |
| **Cron** | 每一页都必须有微张力——但"微"的前提是一个完整的宏观结构在支撑 |

**补充：单元大小的选择权给用户**

用户在不同场景下需要的单元大小不同：
- **手机随手用（Coze Bot）：** 800-1200 字合适。用户在地铁上能读完，有一种"看完一段"的满足感
- **桌面端（MCP Server）：** 可以扩展到 2000-3000 字。用户有更长的注意力窗口

建议在收敛时让用户选"短篇（800字）"或"标准（2000字）"，或者 Bot 根据上下文自动判断——用户在手机上用 Coze 时自动选短篇，在桌面用 Claude Desktop 时自动选标准。

## 建议④⑤：留发散口子 + 继续延展

**你的说法：** 故事单元形成后保留未闭合线索，用户可以选择继续往下走。

**学术依据：✔ 这是最符合叙事本质的设计——故事是流动的，不是一次性的**

| 来源 | 为什么支持 |
|------|----------|
| **Dan Harmon Story Circle** | 8 步循环——故事不是线性的，是循环的。一个单元的"归"和"变"应该是下一个单元的"你"和"需要" |
| **Yorke 辩证结构** | 正→反→合→超越→余韵。一个单元的"余韵"就是下一个单元的"正题" |
| **网文行业** | 连载网文的核心机制就是"钩子->释放->下一个钩子"。每一章结尾都要有一个"为什么" |
| **MICE Quotient**（Card） | 嵌套规则——打开的顺序决定了关闭的逆序。如果在世界型中嵌入了问题型，必须先关闭问题型再关闭世界型 |
| **CK 创意理论** | 最有趣的创意产生于两个已有概念的交叉点。多个故事单元的交叉比单一线性推进更具创意潜力 |

**当前的实现已经很完整了：**

converge 工作流输出带"未闭合线索"和"可发展的方向"，extend 工作流承接上一单元的线索，生成下一段。system prompt 中明确写了"完成后主动问用户继续还是计入画布"。

**一个可以改进的点：**

**发散口子不应该只在故事单元结尾出现，** 还应该在故事单元的中间也出现。比如：

> 你在写一个悬疑故事，主角发现了一张旧照片。Bot 可以说："这个线索有两个方向可以走——照片里的人可能是死去的父亲，也可能是一个完全不同的人。你想让读者猜测哪一个？"

这不是"先写单元再留口子"，而是在写的过程中就建立分叉的可能性。这更接近 CreAgentive 的故事原型双图——角色图 + 情节图解耦叙事逻辑与文本生成，让故事可以有多个可能的走向。

## 建议⑥：故事画布

**你的说法：** 用户可以选择"计入画布"，之后可以随时回来继续。

**这个能力超出了当前 Coze Bot 的工作流范围，因为：**

| Coze Bot 能做什么 | Coze Bot 不能做什么 |
|-----------------|-------------------|
| 在对话中维护当前的"画布"上下文 | 跨会话持久化存储——Coze 变量有限 |
| 在当前对话中展示所有画布内容 | 在另一个会话中恢复画布（除非用 Coze 变量） |
| 在当前对话中延展故事线 | 可视化展示（没有前端） |

**Cross-session 故事画布的最优方案是 PWA + localStorage：**

用户当前会话生成的所有故事单元 → 复制到 PWA → 存入 localStorage → 下一次使用时从 PWA 读取 → 粘贴回 Coze Bot 继续。

这不是技术缺陷，是 Coze Bot 平台的自然限制。任何云端 agent 平台都不能读写用户本地文件。

**如果你需要真正的跨会话故事画布，完整路径是：**

```
Coze Bot（手机）→ 生成故事单元
       ↓ 用户复制
PWA 故事簿（手机/桌面）→ localStorage → 时间线展示
       ↓ 用户从 PWA 复制画布摘要
Coze Bot → 用画布摘要继续延展
```

PWA 中已经有导出/导入功能，也可以将来接 CloudBase 做自动同步。

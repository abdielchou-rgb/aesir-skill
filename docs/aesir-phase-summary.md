## 一、知识库体系

本项目的知识基础来自四个交叉领域，每次设计决策都建立在这些知识的交叉点上。

### 1.1 叙事学（学术界的结构分析传统）

| 来源 | 核心贡献 | 在项目中的用途 |
|------|---------|--------------|
| **Propp · 31 种故事功能** (1928) | 所有故事可分解为有限的功能序列 | 原子库基础层的设计逻辑：不是"原子是词汇"，而是"原子是功能位置" |
| **Campbell · 英雄之旅** (1949) | 跨文化神话的元叙事结构 | 原子组合的语法参考：世界线生成器的底层编排逻辑 |
| **Booker · 7 种基本情节** (2004) | 所有叙事可归为 7 类元情节 | 类型层原子库的分类框架 |
| **Polti · 36 种戏剧情境** (1895) | 人物关系排列产生所有戏剧冲突 | 冲突类型原子的来源 |
| **Greimas · 行动素模型** | 叙事可还原为 6 个功能位置的互动 | 人物关系原子建模的理论依据 |
| **Todorov · 三态模型** | 平衡→破坏→新平衡是一切叙事的底层 | 收敛引擎的"叙事完整性"判断标准 |

## 二、设计目标

### 核心命题

> **让每个人都能用一个模糊的想法，随时随地生成有趣的故事，并且系统越用越懂你。**

### 五条具体目标

| # | 目标 | 对应用户诉求 | 非目标（明确不做） |
|---|------|------------|-------------------|
| 1 | **零门槛启动**：用户说一句话就能开始，不需要学习 | "我就是有个模糊的想法" | 不做长篇小说的章节管理 |
| 2 | **涌现优先**：系统不替代用户创作，而是用发散-收敛循环激发用户自己的创意 | "帮我打开思路，不是帮我写" | 不做"一键生成完本小说" |
| 3 | **越用越对味**：风格档案 + 行为蒸馏让系统在 3 次使用内明显"更懂你" | "它知道我喜欢什么样的故事" | 不做千人一面的通用生成 |
| 4 | **手机随时可用**：PWA 优先，5 秒内完成一次灵感记录 | "等咖啡的时候能用" | 不做桌面端先行的重应用 |
| 5 | **可分享、可导出**：每个故事片段和风格档案都是可传播的社交资产 | "我想让别人看到我的创意" | 不做封闭的内容花园 |

## 三、设计思路

### 哲学一：约束产生涌现，工具不替代人

系统不替用户写，而是提供足够多的可能性（5-8 条世界线），用约束而非填空来引导，保留 30-50% 的"涌现间隙"——生成的片段不说完、不说满，留给用户自己去填充和修改。每轮收敛标配输出"破坏约束版本"。

### 哲学二：三次驯化，从不空手而来

前三次使用强制走"对话优先"路径。第一次：系统只提问不发散。第二次：基于第一次的回答生成 3 条世界线。第三次：开始全功能发散并说明"这是根据你的偏好调整过的"。设计意图是让用户在充分表达后再看到输出，减少冷启动时的认知负担。

### 哲学三：数据资产沉淀，而非消费功能

用户离开系统的迁移成本不是"这个功能别处没有"，而是"我的风格档案在这里"。每一次选择、每一次蒸馏、每一次修改都在塑造个人的叙事风格档案。风格档案可以导出，但导出后失去的是"下次生成的上下文关联"——这才是真正的锁定。

### 哲学四：原子由系统推导，不凭感觉预设

三层原子库：基础层（30-50 个跨文化基元，来自叙事学经典）、类型层（每类型 20-30 个特色原子，从同类型文本中蒸馏提取）、用户层（从用户个人语料和行为中提取）。原子库不是封闭的——用户可以蒸馏、组合、分享。

### 4.2 核心数据结构

| 实体 | 关键字段 | 用途 |
|------|---------|------|
| `Fragment` | id, content, type, stage, createdAt | 创意碎片，用户的原始输入素材 |
| `Worldline` | id, premise, tone, conflict, protag, hook, liked | 一条完整的世界线方向 |
| `Atom` | id, name, desc, category, level | 叙事原子（主角/冲突/调性/场景/转折） |
| `StyleProfile` | tones{}, conflicts{}, total | 用户风格偏好累计 |
| `Distillate` | source, surface, structural, merged_profile | 蒸馏产物（从文本中提取） |

### 4.3 用户核心流程

```
模糊想法 → (前3次: 3轮澄清对话) → 发散引擎 → 5条世界线
  → 用户选择1条 + 约束方向 → 收敛引擎 → 故事片段 + 破坏约束版本
  → 保存到存档 / 再织一个 → 风格档案更新 → 下次发散的隐性约束
```

## 六、当前状态评估

### 6.1 已就绪的功能

- ✅ **完整发散-收敛循环**：模糊想法 → 5 条世界线 → 用户约束 → 故事片段 + 破坏约束版本
- ✅ **前 3 次驯化对话**：自动识别新用户，3 轮对话后生成更准确的世界线
- ✅ **演示模式动态引擎**：无 API key 也可完整体验，输出基于用户输入动态生成（非硬编码）
- ✅ **风格档案**：自动记录调性偏好、冲突偏好，在"我"页面展示
- ✅ **存档管理**：故事片段可保存、按时间轴浏览、展开详情
- ✅ **蒸馏管道（Python）**：文本上传 → 表层 + 中层特征提取 → 风格档案增量
- ✅ **Coze Bot 配置**：可直接导入 Coze 平台
- ✅ **PWA 可部署**：manifest + service worker + vercel 配置就绪

### 6.2 未实现/待完善

- ⬜ **MCP Server 封装**：当前 PWA 和 Python 管道分离，未封装为统一的 MCP 协议服务
- ⬜ **用户认证系统**：无账号，无跨设备同步
- ⬜ **发酵池**：未选世界线的跨会话存储和唤醒（当前每次重新发散，不记忆未选分支）
- ⬜ **蒸馏管道前端集成**：用户上传文件到前端后自动调用蒸馏 API、写回风格档案
- ⬜ **LLM 代理部署**：Suapbase Edge Function 已写好代码，未部署到正式环境
- ⬜ **原子库抽象化**：当前原子是代码中硬编码的 JSON 数组，未做成可热更新、用户可自定义的模块
- ⬜ **风控和计费**：免费版 30 次/月的限制和订阅付费逻辑

### 6.3 已知问题

- 演示模式的动态生成虽然比硬编码好，但前提句模板只有 8 个变体、钩子模板只有 5 个变体——随着使用次数增加，用户会看到重复模式。需要接入 LLM 后才能真正解决
- 前 3 次驯化的 3 个问题当前是通用模板（基于关键词填充），不是真正的 LLM 生成。跟演示模式的同一个问题：模板有限
- 风格档案只记录了调性和冲突的频次分布，没有提取更精细的"句长偏好""视角距离""对话比例"等特征——这些需要接入蒸馏管道后的增量更新

### 关键决策点

| 决策 | 选项 | 建议 |
|------|------|------|
| API key 策略 | 用户自备 key / 系统托管 key | **用户自备 key（当前设计）**：降低合规风险和成本，适合初期验证。后期商业版再走托管 |

| 风险 | 概率 | 影响 | 缓解策略 |
|------|------|------|---------|

## 八、总结

从第一轮关于"好故事是什么"的哲学讨论开始，经过叙事学的结构分析、学术界 vs 商界的方法论对比、还原论与涌现的张力辩论、好莱坞 vs 网文的实践知识拆解，这个项目走过了从认知到工程的完整路径。

到目前为止的交付物形成了一个最小的可验证系统：用户可以在手机浏览器中打开一个页面，输入一个模糊想法，经过 3 轮对话（前 3 次）或直接发散（第 4 次开始），看到 5 条真正不同的世界线，选择一条并加约束，得到一个故事片段。系统会自动记录她的风格偏好，并在下一次使用中体现。

这不是一个完成的产品——它没有用户系统、没有后端、发酵池还需要基础设施——但它的核心交互模式已经跑通，并且有了一条清晰的演进路径：先上线验证 → 再完善打磨 → 后商业化。
